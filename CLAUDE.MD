# FIRST ACTION OF EVERY SESSION — no exceptions

**Read `STATUS.md`. State the current tier and any blocking items out loud before touching anything.**

Do not proceed on assumed context from a previous session. The methodology drifted out of sync with
reality once already — verify-wall compression and oracle breadth were prescribed for many ticks and
simply had not been done, while backlog work carried on, and nobody noticed until it was explicitly
asked about. `STATUS.md` exists so that gap is visible in five seconds instead of requiring an
interrogation.

## FIRST PRINCIPLE — THE RATCHET: bank additive step-changes, never regress

> **Every tick must leave the browser strictly more capable than it found it, and nothing that worked
> before may work less well after. Progress is a RATCHET: it only turns one way.**

This is the north star that sits *above* the three bars, and it is what the whole mechanism stack
exists to enforce. It has three faces, and **all three are non-negotiable**:

| Never regress | Measured by |
|---|---|
| **Capability** — a class of site that rendered must not stop rendering | the gate wall (`verify.sh`), the oracle crawl, WPT |
| **Performance** — a page that was fast must not get slow | F1/F2 perf floors, the crawl's own clock |
| **Instrument fidelity** — a number that was honest must not start lying | `falsify.sh`, the accounting reconciliations, `NO_REPORT`-style self-guards |

**"Additive" is the operative word.** The unit of progress is not a diff, it is a **qualitative
capability step-change**: *"this class of the web now works."* `<iframe>` renders. `:has()` matches.
`load` fires. `click()` activates. Each is banked — gated, falsified, journalled, committed, pushed —
and **cannot silently come back**, because a gate that has been proven to go red is a ratchet tooth.

**The trajectory this has actually produced, and why the discipline is the reason:**

> nothing but Wikipedia, crashing elsewhere on duplicated cascades → dedup + lean efficiency →
> parity on 20 doc-web sites → SPA/app-web support → the 265-site differential oracle →
> **WPT in the mechanical pipeline** → categorical support of the doc web and app web →
> *(next)* the **platform web**: lazy-loaded feeds, iframes, media.

**A tick that produces a better description of the same browser is a tax, not a turn of the ratchet.**
A tick that buys capability by losing performance, or buys a number by degrading an instrument, is
**not additive — it is a trade**, and trades are refused. If a change cannot be shown to be strictly
additive on all three faces, it is not done.

## INSTRUMENTS HAVE A DONE. THE BROWSER IS THE POINT.

An instrument is **finished** when it (a) reports every input, (b) does not collapse different findings
into one word, and (c) produces a ranked work list. **At that moment you stop working on it and start
working through its output.** WPT reached that state in tick 43: `NO_REPORT 0`, `HANG`/`SLOW`/
`TH_TIMEOUT` disambiguated, per-directory ranking. **It is done. Do not polish it.**

> **The goal is a daily driver that handles basically the whole internet** — the doc web, the app web,
> and the platform web — with **qualitative capability step-changes** and performance that beats
> Chromium. The oracle and WPT exist **only** to say *which capability to build next*. They are not
> the work.

**The failure mode to refuse by name: epistemological fog.** Endlessly refining how we measure, so that
every tick produces a better *description* of the same broken browser. If a tick's output is a more
accurate number and no new capability, **that tick was a tax, not progress** — it is justified only when
the number was actively *lying* (as it was once, and that is now fixed).

**The test for any proposed instrument work:** *"does this change which capability I build next?"* If
no — **it is not the tick.** Go build the capability.

## RELEASE CADENCE — ship on a CAPABILITY STEP-CHANGE, not on a calendar and not every tick

> **A version number should mean something changed for the user.**

**The trigger is mechanical, and it reads the loop's own ledger** (`.github/workflows/release.yml`):

- a tick whose commit declares **`TICK SHAPE: pattern-class`** (or `capability`) **and moves
  `docs/loop/WEB-PATTERNS.md`** — which is this project's *definition* of *"a class of the web now
  works"*;
- **plus** a green verify wall on exactly that tree (the pre-commit hook already guarantees it);
- **or** a manual `workflow_dispatch`, for a reason a machine cannot see.

**Everything else does NOT cut a release** — infrastructure, instrumentation, CI, refactors, docs. That is
the *point*, not an oversight: this project just spent **six consecutive ticks on CI**, and **no user
would ever want a binary of that.** Shipping every tick floods the tags with artifacts nobody can use;
shipping on a timer decouples the artifact from the reason to want it.

**The version is derived, never hand-typed:** `v0.<TICK>.0`, read from `STATUS.md`. **The tick number IS
the version, because the tick IS the unit of progress here.**

**The release notes are the capability ledger's newest section, verbatim.** It is already written in the
only terms a release note should be written in — *"this class of the web now works"* — so there is nothing
to compose and nothing to embellish. *If the ledger entry is not worth reading, the release was not worth
cutting.*

**The artifact is the headless renderer** (`manuk render <url>`) built static for Linux/macOS/Windows: no
display, no GPU, no JS engine — the one build that genuinely runs anywhere.

## COMMIT AND PUSH CADENCE — the interval is the TICK, and it is not negotiable

- **Commit exactly once per tick, at the tick boundary.** The commit is what makes a tick *exist*. It
  is admissible only when all three hold: **`verify.sh` green on exactly this tree** (the receipt is
  tree-hashed — "the gates passed" cannot be a memory of an earlier version of the diff), **a journal
  entry with a RESULT**, and **a `WIKI:` trailer**. The hook enforces all three.
- **Never commit mid-tick on `main`.** A half-tick commit is a claim about a gate that has not run.
  If a tick is long, checkpoint on a **branch**, never on `main`.
- **Update `README.md` every tick, as appropriate.** It is the only file an outside reader opens, and a
  README that describes an engine two months stale is a *claim*, not documentation. If the tick changed
  what the engine can do, what it measures, or how it is run — the README says so before the commit.
- **Push after every tick commit.** The loop runs unattended for hours; the remote is the only copy
  that survives a machine loss. **Pushing is cheap. Re-deriving a day of ticks is not.**
- ⚠ **NEVER `git checkout <file>` or `git stash` a file with uncommitted work to "test a hypothesis".**
  This is not a style preference — **tick 43 destroyed real, working code this way**, reverting two
  files to chase a "regression" that turned out to be a *misconfigured test command* (the gate had been
  run without `--features spidermonkey`, so the engine had no JS at all and every script "failed").
  **Bisect by copying the file aside, or by commenting code out — never by reverting the only copy.**
  And see PROCESS: *a RED verdict is a claim too, and destructive action on an unverified claim is how
  work gets lost.*

## THE STANDING OPERATING LOOP — adopt as written

**Work perpetually on the tick/eval loop. Do not hand back.** Not at a milestone, not when a tick feels
complete, not when unsure what is next — **the ledgers say what is next.** Only an explicit instruction

# CLAUDE.md — Browser Engine Project

Project directive for Claude Code. Read this in full before making changes. This file governs scope, architecture, and — critically — which parts of this codebase are safe to modify freely versus which require explicit human sign-off regardless of how confident the change looks.

## Current phase: shared engine core, with both front-ends live

Two front-ends now sit on one shared engine core:

1. **Headful, human-facing browser** (`shell`) — a GUI app with tabs, navigation, and rendering (headless `render` to PNG + interactive `winit`/`wgpu` `browse` window).
2. **Headless agentic browser** (`agent`) — a page driver an LLM can operate (navigate / click / scroll / screenshot / observe), driven by a provider-agnostic inference backend. This phase was explicitly authorized and is now in an initial, working form.

The common engine core (networking, parsing, CSS, DOM/JS, layout, text) is realized in the `engine/*` crates and the shared `engine/page` pipeline; the two front-ends diverge only at how they consume a rendered page (window present vs. screenshot+read). Keep that split clean — do not wire front-end-only assumptions into the parse/layout/JS layers.

`.env` holds the agent phase's provider credentials and is gitignored; `.env.example` documents the shape. The agent's inference backend is decoupled behind an `InferenceBackend` trait, so a provider is one implementation and never a dependency of the agent logic — local `llama-server`, Ollama and any OpenAI-compatible endpoint are equally first-class.

## Current state (implemented)

A working foundation, not a finished browser. What exists and is verified:

- **Networking** (`engine/net`): pooled `hyper-util` client over a `hyper-rustls` connector (no OpenSSL), **HTTP/2 auto-negotiated** (ALPN), **connection reuse**, **gzip/br/deflate decoding** (`async-compression`), redirects, and a general `request` (POST) reused by the agent's Groq client. Live-fetches real sites. *(P0.4 landed.)*
- **Parse → render pipeline** (`engine/{html,dom,css,layout,text,paint}` via `engine/page`): `html5ever` → arena DOM → minimal CSS cascade (selectors/specificity/inheritance) → from-scratch block + inline layout (wrapping, centering, **adjacent-sibling margin collapsing**, **floats** via a BFC-aware `FloatContext` with clearance/shrink-to-fit/text-flow-around, **relative/absolute/fixed positioning** against the containing-block chain, **tables** with fixed/auto column sizing; `taffy` for flex) → `fontdb`/`fontdue` text → `tiny-skia` CPU raster to PNG. Renders real pages (example.com and local pages) end to end.
- **JS** (`engine/js`): `JsRuntime` trait with a no-op default; **SpiderMonkey** (`mozjs`) behind `--features spidermonkey` **actually evaluates JS**. Process-global engine, shared across isolates. Modification boundary respected (see below).
- **CSS engine seam** (`engine/css`): minimal cascade default; **Stylo** behind `--features stylo` (links and builds; adapter currently delegates).
- **Compositor** (`engine/compositor`): tab render/memory tiers (focused-GPU / background-CPU / hibernated), damage tracking, scroll.
- **Headful shell** (`shell`): `render <url>` (headless PNG) and `browse <url>` (winit/wgpu window presenting the CPU raster).
- **Headless agent** (`agent`): `AgentBrowser` driver, `InferenceBackend` trait (multimodal), `run_task` loop, `GroqBackend`. Live-tested on Groq `qwen/qwen3.6-27b`: text extraction, link comprehension, link navigation, and multimodal screenshot reading all pass.
- **Conformance** (`tests/wpt`): built-in layout reftests run today; the upstream WPT runner is the growth path. 40 workspace tests pass; zero warnings.

## Mission

Build a browser engine, from first principles in Rust, that:
- Targets the **broadest possible set of real-world websites**, with the explicit exception of the smallest tier of adversarially hostile sites (aggressive bot-wall/fingerprint-gated sites like LinkedIn, X, Indeed — these are out of scope for the core engine and would be handled by narrow, separate adapters in a different phase, never by chasing generic compatibility with their defenses).
- Is verified against **Web Platform Tests (WPT)**, not against matching Chromium's specific undocumented quirks. Standards compliance is the compatibility target. Do not write code whose purpose is to replicate a Chromium-specific non-standard behavior — if a target site depends on that, it's in the excluded hostile/non-standard tier.
- Is optimized from first principles for binary size, per-tab memory, GPU-quality rendering, and click-to-navigate latency — not merely "acceptable," but genuinely disruptive relative to Chromium/Firefox on these dimensions for the sites in scope.
- Runs identically on Linux, macOS, and Windows with no platform serving as the "primary" target and others as afterthoughts.

## Architecture — the stack

Reuse mature components where reinventing them adds security/correctness risk with no differentiation value. Build from scratch, verified against WPT, where the target site set is bounded and a from-scratch build is both tractable and gives real performance upside. Do not blur this line without discussion — see "Modification boundaries" below.

| Layer | Approach | Components |
|---|---|---|
| Networking | Reuse | `hyper` (HTTP/1.1, HTTP/2), `quinn` (HTTP/3/QUIC), `tokio-tungstenite` (WebSocket), `rustls` (TLS — pure Rust, no OpenSSL system dependency), `tokio` async runtime |
| HTML parsing | Reuse | `html5ever` |
| CSS parsing + cascade | Reuse | Stylo (Servo's/Firefox's production CSS engine) |
| Layout | Reuse + build | `taffy` for flexbox/grid; custom block, inline, table, positioned, and float layout built from scratch, verified against WPT layout reftests |
| DOM + Web API surface | Build | Custom, spec-driven (WebIDL), verified per-API against its WPT test suite. This is the largest-volume component — budget accordingly |
| JS engine | Reuse, do not reimplement | SpiderMonkey, embedded via `mozjs` / `mozjs-sys` Rust bindings (the same integration path Servo uses). **Not V8.** SpiderMonkey is C++ and Rust (Rust in the parser layer only — the JIT and GC are C++); this is not a "pure Rust" engine, it is a mature C++ engine with a proven Rust FFI layer |
| Text shaping | Reuse | Parley + swash + fontdb |
| Paint / rasterization | Reuse | Vello — GPU-compute renderer via `wgpu`, with Vello CPU / Vello Hybrid as lighter fallback tiers for background/hibernated tabs. **Currently alpha upstream** — prototype against actual target rendering needs (gradients, blur, complex clips) before committing further |
| GPU abstraction | Reuse | `wgpu` — Vulkan (Linux/Windows), Metal (macOS), DX12 (Windows). No Linux-desktop or specific graphics-package assumptions |
| Windowing / event loop | Reuse | `winit` |
| Compositor | Build | Layer compositing, scroll, damage tracking, animations — built on Vello + wgpu |
| Concurrency/process model | Build | **REVISED — one OS process per tab, SpiderMonkey embedded in each** (`docs/loop/PROCESS-MODEL.md`). The former "compartments multiplexed over a few processes — *not* one process per tab" is **superseded**: in-process containment of a memory-corruption fault is not achievable in *any* production C++ JS engine, so the process boundary **is** the containment. **Per-*origin* Site Isolation is REJECTED** (Chromium names it as its own primary memory-bloat driver). The lean-memory claim is defended by **tiered hibernation**, not by fewer processes: at 100 tabs the question is *how many tabs have a live process at all*, not how small a live tab is. |

## Compatibility verification methodology

- Ground truth is **WPT (Web Platform Tests)** — the cross-browser conformance suite used by Chromium, Firefox, WebKit, and Servo themselves. Every layout, CSS, and DOM/API feature you implement should be checked against the relevant WPT subset before being considered done.
- Do not hand-roll a compatibility test corpus from scratch as the primary verification method — WPT already exists and is comprehensive. Use targeted real-site screenshot diffing (against a real Chrome render) as a secondary check for visual fidelity, not as the primary conformance signal.
- When a feature is ambiguous or under-specified even in WPT (this happens — floats and table layout are notorious), document the interpretation chosen and why, rather than silently picking one.

## Modification boundaries — read before touching the JS engine

This is the most important section in this file. Violating it is the one mistake in this project with real security consequences.

**Safe to modify, verify by benchmark or binary size:**
- SpiderMonkey build configuration and feature flags (stripping unused WASM support, the debugger/inspector protocol, unused ICU locale data — all legitimate binary-size wins)
- The FFI/binding layer between Rust and SpiderMonkey (`mozjs` usage, custom bindings) — this is well-specified, low-blast-radius, and the correct place for iteration
- Startup snapshot tuning (shared heap contents for fast isolate startup)

**Do not modify without dedicated fuzzing infrastructure and explicit sign-off:**
- SpiderMonkey's JIT compiler internals (WarpMonkey/IonMonkey) or garbage collector internals
- Any change to how untrusted JavaScript is sandboxed or isolated

Reasoning, so this isn't just an arbitrary rule: JIT miscompilation bugs — incorrect type or escape-analysis assumptions in the optimizing compiler — are historically the largest single source of exploitable browser RCE vulnerabilities. The reason SpiderMonkey and V8 are trustworthy today isn't just that the code is correct, it's that the code has survived years of adversarial fuzzing (OSS-Fuzz/ClusterFuzz) that this project has no equivalent of. Faster code generation does not substitute for that calendar-bound adversarial exposure. If a task seems to require touching this layer, stop and flag it rather than proceeding — this is a "come back to the human" boundary, not a "do it carefully" one.

## Performance targets

- **Binary size**: static linking + LTO + stripped symbols; `panic = "abort"` in release profile; musl target on Linux for a fully static binary with no dynamic-link bloat; static CRT linking on Windows to avoid requiring a redistributable runtime.
- **Per-tab memory**: isolate-based tabs, not process-based. Background/hibernated tabs should freeze JS execution and evict to disk, dropping to Vello CPU or no active renderer at all — full Vello GPU + active JS is reserved for the focused tab only.
- **Click-to-navigate latency**: streaming/incremental parse and layout (begin building the DOM and layout tree as bytes arrive, don't wait for the full response); zero-copy handoff from socket buffer to tokenizer where feasible; speculative preconnect/DNS-prefetch on link hover.
- **Rendering quality**: Vello + wgpu is the graphics-quality lever — this is a from-scratch, GPU-compute-first architecture, not a CPU/GPU hybrid retrofit. Treat visual fidelity and frame timing as first-class, measured metrics, not an afterthought to functional correctness.

## Portability

- Default async I/O path (`tokio`'s standard reactor) must work identically on Linux, macOS, and Windows. Do not introduce Linux-only dependencies into the core engine.
- `io_uring`-based optimization (`tokio-uring` or similar) is Linux-only and is a *future, optional* fast path relevant to the eventual agent/headless phase, not this one. Do not add it to the human-operator GUI codebase now — the concurrency profile doesn't need it here.
- No Linux-desktop assumptions (no GTK/Qt bindings, no specific display-server assumptions) — `winit` and `wgpu` already abstract this; keep it that way.

## Non-goals

- No adapters for LinkedIn, X, Indeed, or other bot-walled sites; no bot-evasion in the core engine.
- No pixel-parity chase with Chromium beyond what WPT compliance already gets you.
- No DRM/EME.
- No horizontal multi-node scaling concerns — this is single-binary, single-machine scope.
- The agent must not couple to a specific inference provider — every provider is one `InferenceBackend`, never a dependency of the agent logic. **Credentials and local-only development harnesses are never committed** (see `.gitignore`); a committed tool must be runnable by anyone who clones this repository.

**Research dispositions (2026-07, primary sources cited inline):**

- **E6 bring-your-own-key in-browser AI — PASS-2 RESOLVED (security model: layered CaMeL structural separation + OWASP Action-Guard).** *Finding:* the panel is architecturally trivial — **reuse the `InferenceBackend` abstraction**; the headful AI panel and the headless agent are *the same core* with different front-ends (the project thesis). - **E7 profiles / containers / VPN.** *Finding:* Firefox distinguishes **profiles** (complete separation of *all* data — bookmarks/passwords/settings/extensions/history/cookies) from **containers** (Multi-Account: separate only cookies/logins/site-data *within* a profile — a contextual-identity cookie-jar partition), with **Total Cookie Protection** = a per-website cookie jar (partitioned storage) (support.mozilla.org containers/profiles/TCP). SOCKS5/HTTP proxy is application-level, unencrypted routing; `tokio-socks` is the mature async Rust client; `wireproxy` even exposes WireGuard *as* a SOCKS5 proxy — so "VPN" can reduce to "point at a proxy" (github.com/sticnarf/tokio-socks; github.com/windtf/wireproxy). *Disposition:* **confident, lean framing.** Profiles/containers = a **storage-partition** design (profile ⊃ container jar ⊃ per-site jar/TCP) — **hand-roll the partition key** over the storage layer, but **defer, blocked on a storage layer** (we have none: no cookies/cache/history/localStorage yet). **VPN = reuse SOCKS5/HTTP proxy** (`tokio-socks`) so the browser routes through a user-provided proxy (WireGuard via `wireproxy` externally); **bundling a WireGuard/OpenVPN client is out of scope** (scope-heavy, no strong reason) — the RESEARCH-recommended lean framing. *Roadmap:* §9 (profiles deferred on storage; SOCKS proxy in §2 net); no WPT target. **E7 LANDED (2026-07-10, `engine/net::{cookies,storage,proxy}`, default build).** The item's stated prerequisite — "begins by building the storage layer" — is honored: **(1) `cookies` = an RFC 6265 jar.** Parses `Set-Cookie` (Domain/Path/Secure/HttpOnly/SameSite/Max-Age/Expires incl. an IMF-fixdate parser verified against the RFC's canonical `784111777`), and enforces the rules that carry security weight: **`Domain` may not be a public suffix** (`Domain=com` / `Domain=co.uk` rejected via `psl` — the classic supercookie hole; unknown suffixes **fail closed**), `Domain` must domain-match the request host, **host-only** cookies never reach subdomains, `Secure` cookies never go over `http`, **an insecure origin may not *set* a `Secure` cookie** ("Leave Secure Cookies Alone" — else plaintext http overwrites the https session), `Max-Age` beats `Expires` (§5.2.2), path-matching respects segment boundaries (`/application` ∌ `Path=/app`), and send-order is longest-path-then-oldest (§5.4.2). **(2) `storage` = the profile ⊃ container ⊃ top-level-site partition.** Two containers keep **separate jars for the same site** (the item's headline acceptance); profiles are fully isolated; and keying on the *top-level site* gives **unconditional Total Cookie Protection** — the same tracker embedded on `news-site.org` and `shop-site.com` gets two different jars and cannot join the visits. `localStorage` is partitioned by (partition, script origin); history is per-**profile** (survives a container reset, the Firefox model); `clear_container`/`clear_profile` implemented. **`SameSite` is enforced here**, not in the jar, because it needs request context — and the *asymmetry is implemented correctly*, which is the easy thing to get wrong: a **subresource** is cross-site vs. `top_level`, but a **top-level navigation** must be judged against the **initiator**, since the destination *becomes* the top level. - **G-d deterministic replay / provenance.** *Finding:* the established technique for reproducible agents is an **append-only event log recording model + tool responses** (models aren't reproducible, so you record their outputs), with strict-mode event-by-event divergence checking as a *proof* of reproducibility, and "decision-complete provenance" enabling re-evaluation of historical runs under new policy without re-execution (arXiv:2605.21997 "The Log is the Agent"; arXiv:2505.17716 record&replay). Crucially, **GPU rendering and layout are documented sources of replay-nondeterminism** (WebKit "Web Replay"; GPU kernel autotuning) — so GPU/cloud-VM agents diverge on replay. *Disposition:* **confident — real utility (auditable/reproducible agent runs for compliance/research/debug/regression), uniquely well-positioned.** Our agent is **display-free with a deterministic CPU raster** (`tiny-skia`) → stable screenshot hashes make exact replay achievable where GPU agents can't; the clean `InferenceBackend` seam records model I/O. **Reuse** the existing `AgentOutcome` transcript → extend to a full event log; low marginal cost. *Roadmap:* §10 (+ §4 agent, §5 correctness — replay doubles as a regression harness). No WPT target. **G-d LANDED (2026-07-10, `manuk-agent::replay` + `run_task_recorded`).** An agent run is reproducible only if you record the parts that are not: the **model** (same prompt, different sampling) and the **network**. So the append-only log stores the model's raw replies **verbatim**, and a replay feeds them back in order via `ReplayBackend` instead of calling a model at all. Everything else — parse/style/layout/CPU raster — is deterministic, which is exactly why this works: the agent renders with `tiny-skia` on the CPU and never touches a GPU, so **screenshot digests are stable** where a GPU-backed agent's drift across machines and drivers (pinned by `cpu_raster_screenshot_digest_is_stable_across_identical_renders`). *Log:* `Event::{Start,Observed,Model,Acted,Blocked,Finished}` serialized as **JSON Lines**, so it can be appended and tailed and a **truncated** log still parses up to the last whole line (tested); a *malformed* line is an error naming its line number, never silently dropped (a dropped line would corrupt replay order). `ObservationRecord` stores url/title/**text digest**/the full a11y tree/**screenshot digest**/scroll — the tree in full because it is small, semantic, and the most useful thing to read when a replay diverges. *Replay:* `check_step` compares recorded vs freshly-computed observations; `Strict` aborts on the first divergence and **names the differing field**, `Lenient` collects them; `ReplayBackend` erroring when the log is **exhausted** rather than stopping silently. **A green strict replay IS the reproducibility proof**, and the same machinery doubles as a regression harness (§5). *Digest note:* `digest()` is **FNV-1a — a non-cryptographic checksum** used only to compare two renderings; it is never used for security, so CLAUDE.md's hand-rolled-crypto prohibition is untouched (E2's at-rest encryption still uses audited AEAD crates). *Efficiency:* the per-step screenshot taken for the record is **reused** as the multimodal message content rather than painted twice. **Verified end-to-end:** a scripted 2-step run is recorded, JSONL-round-tripped, and **replayed strictly with zero divergences and every recorded reply consumed**; replaying against a *changed* page correctly **diverges** at step 0. 40 agent tests green; workspace green (38 ok blocks); clippy-clean. **Documented gap (not faked):** the **network is not recorded** — replaying against a live site that has changed will (correctly) report divergence. The log proves what was seen; it does not resurrect the server. Recording a response cache is the tracked follow-up. Platform: [XP]; verified on Linux.
- **G-e instant per-tab resource honesty.** *Finding:* real pain (the founding "why is my fan spinning") but **not novel** — Chrome/Firefox already ship per-tab/per-process task managers + `about:memory`. The genuine wrinkle is **ours specifically**: our **isolate-per-tab-in-shared-process** model (C1) means the OS won't hand us per-tab CPU/RAM for free (unlike Chrome's process-per-tab). SpiderMonkey **does** support per-compartment/zone memory attribution (Firefox's `about:memory` reports per-JS-compartment; every GC cell belongs to ≤1 compartment) — but with a caveat that ~30–45% lands in "heap-unclassified" (imperfect attribution) (firefox-source-docs js/gc; blog.mozilla.org nnethercote). *Disposition:* **confident — do it, reuse SpiderMonkey per-compartment memory reporters** for per-tab JS heap + **hand-roll** compositor per-tab layout/paint/tile accounting + a shell task-manager UI. It rides on the same instrumentation **C1 + §8** need; the shared-process **self-attribution** (esp. per-tab *CPU*, which needs self-timing with no per-process split, and the heap-unclassified gap) is the wrinkle to solve there, not a blocker. *Roadmap:* §10 (+ C1, §8, §1). No WPT target. **G-e LANDED (2026-07-10, `shell::tab::{ResourceReport,TabResource}` + Ctrl+M in the GUI).** *Honesty* is the whole item, so every number is labelled for what it is: `retained_bytes` is an explicit **proxy** (what a discard would reclaim: the `Page`'s estimated heap + retained source HTML), while `process_rss_bytes` is the **OS's real** figure from `/proc/self/status`. ## Working agreements

- Any change touching `/engine/layout`, `/engine/css`, or `/engine/js` should be checked against the relevant WPT subset before being considered complete — cite which tests were run.
- Any change touching binary size or per-tab memory should report before/after numbers.
- Treat SpiderMonkey as a vendored dependency you configure and bind against, not a codebase you patch internals of — see Modification boundaries above.
- When in doubt about whether a site belongs in the "broadest possible, excluding hostile" target set or the excluded tier, default to excluding rather than building bot-evasion logic into the core engine.
- **Documentation is part of "done."** On every major change — a new crate, a new capability, a wired feature seam, or a changed public API — update **both** `README.md` and this file in the same change. `README.md` must describe the stack *as it actually is* (no aspirational claims in the per-crate status; move aspirations to "Directions of improvement" below). Re-read and re-prioritize the "Current state" and "Directions of improvement" sections here periodically.
- The agent's inference is provider-agnostic: keep it behind the `InferenceBackend` trait — every provider is one implementation. **Never commit credentials, `.env`, or local-only development harnesses.** A committed tool must be runnable by anyone who clones this repository; one that depends on a personal environment is a local tool by definition.
