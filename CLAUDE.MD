# CLAUDE.md — Browser Engine Project

Project directive for Claude Code. Read this in full before making changes. This file governs scope, architecture, and — critically — which parts of this codebase are safe to modify freely versus which require explicit human sign-off regardless of how confident the change looks.

## Current phase: shared engine core, with both front-ends live

Two front-ends now sit on one shared engine core:

1. **Headful, human-facing browser** (`shell`) — a GUI app with tabs, navigation, and rendering (headless `render` to PNG + interactive `winit`/`wgpu` `browse` window).
2. **Headless agentic browser** (`agent`) — a page driver an LLM can operate (navigate / click / scroll / screenshot / observe), driven by a provider-agnostic inference backend. This phase was explicitly authorized and is now in an initial, working form.

The common engine core (networking, parsing, CSS, DOM/JS, layout, text) is realized in the `engine/*` crates and the shared `engine/page` pipeline; the two front-ends diverge only at how they consume a rendered page (window present vs. screenshot+read). Keep that split clean — do not wire front-end-only assumptions into the parse/layout/JS layers.

`.env` holds `GROQ_API_KEY` variables for the agent phase; `.env` is gitignored. The committed agent runner uses a **single** `GROQ_API_KEY` (falling back to `GROQ_API_KEY`); the local-only development harness is a **local-only, gitignored** test tool. The agent's inference backend is decoupled behind an `InferenceBackend` trait — Groq is one implementation, not a hard dependency of the agent logic.

## Current state (implemented)

A working foundation, not a finished browser. What exists and is verified:

- **Networking** (`engine/net`): pooled `hyper-util` client over a `hyper-rustls` connector (no OpenSSL), **HTTP/2 auto-negotiated** (ALPN), **connection reuse**, **gzip/br/deflate decoding** (`async-compression`), redirects, and a general `request` (POST) reused by the agent's Groq client. Live-fetches real sites. *(P0.4 landed.)*
- **Parse → render pipeline** (`engine/{html,dom,css,layout,text,paint}` via `engine/page`): `html5ever` → arena DOM → minimal CSS cascade (selectors/specificity/inheritance) → from-scratch block + inline layout (wrapping, centering; `taffy` for flex) → `fontdb`/`fontdue` text → `tiny-skia` CPU raster to PNG. Renders real pages (example.com and local pages) end to end.
- **JS** (`engine/js`): `JsRuntime` trait with a no-op default; **SpiderMonkey** (`mozjs`) behind `--features spidermonkey` **actually evaluates JS**. Process-global engine, shared across isolates. Modification boundary respected (see below).
- **CSS engine seam** (`engine/css`): minimal cascade default; **Stylo** behind `--features stylo` (links and builds; adapter currently delegates).
- **Compositor** (`engine/compositor`): tab render/memory tiers (focused-GPU / background-CPU / hibernated), damage tracking, scroll.
- **Headful shell** (`shell`): `render <url>` (headless PNG) and `browse <url>` (winit/wgpu window presenting the CPU raster).
- **Headless agent** (`agent`): `AgentBrowser` driver, `InferenceBackend` trait (multimodal), `run_task` loop, `GroqBackend`. Live-tested on Groq `qwen/qwen3.6-27b`: text extraction, link comprehension, link navigation, and multimodal screenshot reading all pass.
- **Conformance** (`tests/wpt`): built-in layout reftests run today; the upstream WPT runner is the growth path. 40 workspace tests pass; zero warnings.

## Mission

Build a browser engine, from first principles in Rust, that:
- Targets the **broadest possible set of real-world websites**, with the explicit exception of the smallest tier of adversarially hostile sites (aggressive bot-wall/fingerprint-gated sites like LinkedIn, X, Indeed — these are out of scope for the core engine and would be handled by narrow, separate adapters in a different phase, never by chasing generic compatibility with their defenses).
- Is verified against **Web Platform Tests (WPT)**, not against matching Chromium's specific undocumented quirks. Standards compliance is the compatibility target. Do not write code whose purpose is to replicate a Chromium-specific non-standard behavior — if a target site depends on that, it's in the excluded hostile/non-standard tier.
- Is optimized from first principles for binary size, per-tab memory, GPU-quality rendering, and click-to-navigate latency — not merely "acceptable," but genuinely disruptive relative to Chromium/Firefox on these dimensions for the sites in scope.
- Runs identically on Linux, macOS, and Windows with no platform serving as the "primary" target and others as afterthoughts.

## Architecture — the stack

Reuse mature components where reinventing them adds security/correctness risk with no differentiation value. Build from scratch, verified against WPT, where the target site set is bounded and a from-scratch build is both tractable and gives real performance upside. Do not blur this line without discussion — see "Modification boundaries" below.

| Layer | Approach | Components |
|---|---|---|
| Networking | Reuse | `hyper` (HTTP/1.1, HTTP/2), `quinn` (HTTP/3/QUIC), `tokio-tungstenite` (WebSocket), `rustls` (TLS — pure Rust, no OpenSSL system dependency), `tokio` async runtime |
| HTML parsing | Reuse | `html5ever` |
| CSS parsing + cascade | Reuse | Stylo (Servo's/Firefox's production CSS engine) |
| Layout | Reuse + build | `taffy` for flexbox/grid; custom block, inline, table, positioned, and float layout built from scratch, verified against WPT layout reftests |
| DOM + Web API surface | Build | Custom, spec-driven (WebIDL), verified per-API against its WPT test suite. This is the largest-volume component — budget accordingly |
| JS engine | Reuse, do not reimplement | SpiderMonkey, embedded via `mozjs` / `mozjs-sys` Rust bindings (the same integration path Servo uses). **Not V8.** SpiderMonkey is C++ and Rust (Rust in the parser layer only — the JIT and GC are C++); this is not a "pure Rust" engine, it is a mature C++ engine with a proven Rust FFI layer |
| Text shaping | Reuse | Parley + swash + fontdb |
| Paint / rasterization | Reuse | Vello — GPU-compute renderer via `wgpu`, with Vello CPU / Vello Hybrid as lighter fallback tiers for background/hibernated tabs. **Currently alpha upstream** — prototype against actual target rendering needs (gradients, blur, complex clips) before committing further |
| GPU abstraction | Reuse | `wgpu` — Vulkan (Linux/Windows), Metal (macOS), DX12 (Windows). No Linux-desktop or specific graphics-package assumptions |
| Windowing / event loop | Reuse | `winit` |
| Compositor | Build | Layer compositing, scroll, damage tracking, animations — built on Vello + wgpu |
| Concurrency/process model | Build | Isolate/compartment-per-tab (SpiderMonkey compartments) multiplexed across a small number of OS processes — not one OS process per tab. Share startup snapshots across isolates to cut per-tab baseline memory |

## Compatibility verification methodology

- Ground truth is **WPT (Web Platform Tests)** — the cross-browser conformance suite used by Chromium, Firefox, WebKit, and Servo themselves. Every layout, CSS, and DOM/API feature you implement should be checked against the relevant WPT subset before being considered done.
- Do not hand-roll a compatibility test corpus from scratch as the primary verification method — WPT already exists and is comprehensive. Use targeted real-site screenshot diffing (against a real Chrome render) as a secondary check for visual fidelity, not as the primary conformance signal.
- When a feature is ambiguous or under-specified even in WPT (this happens — floats and table layout are notorious), document the interpretation chosen and why, rather than silently picking one.

## Modification boundaries — read before touching the JS engine

This is the most important section in this file. Violating it is the one mistake in this project with real security consequences.

**Safe to modify, verify by benchmark or binary size:**
- SpiderMonkey build configuration and feature flags (stripping unused WASM support, the debugger/inspector protocol, unused ICU locale data — all legitimate binary-size wins)
- The FFI/binding layer between Rust and SpiderMonkey (`mozjs` usage, custom bindings) — this is well-specified, low-blast-radius, and the correct place for iteration
- Startup snapshot tuning (shared heap contents for fast isolate startup)

**Do not modify without dedicated fuzzing infrastructure and explicit sign-off:**
- SpiderMonkey's JIT compiler internals (WarpMonkey/IonMonkey) or garbage collector internals
- Any change to how untrusted JavaScript is sandboxed or isolated

Reasoning, so this isn't just an arbitrary rule: JIT miscompilation bugs — incorrect type or escape-analysis assumptions in the optimizing compiler — are historically the largest single source of exploitable browser RCE vulnerabilities. The reason SpiderMonkey and V8 are trustworthy today isn't just that the code is correct, it's that the code has survived years of adversarial fuzzing (OSS-Fuzz/ClusterFuzz) that this project has no equivalent of. Faster code generation does not substitute for that calendar-bound adversarial exposure. If a task seems to require touching this layer, stop and flag it rather than proceeding — this is a "come back to the human" boundary, not a "do it carefully" one.

## Performance targets

- **Binary size**: static linking + LTO + stripped symbols; `panic = "abort"` in release profile; musl target on Linux for a fully static binary with no dynamic-link bloat; static CRT linking on Windows to avoid requiring a redistributable runtime.
- **Per-tab memory**: isolate-based tabs, not process-based. Background/hibernated tabs should freeze JS execution and evict to disk, dropping to Vello CPU or no active renderer at all — full Vello GPU + active JS is reserved for the focused tab only.
- **Click-to-navigate latency**: streaming/incremental parse and layout (begin building the DOM and layout tree as bytes arrive, don't wait for the full response); zero-copy handoff from socket buffer to tokenizer where feasible; speculative preconnect/DNS-prefetch on link hover.
- **Rendering quality**: Vello + wgpu is the graphics-quality lever — this is a from-scratch, GPU-compute-first architecture, not a CPU/GPU hybrid retrofit. Treat visual fidelity and frame timing as first-class, measured metrics, not an afterthought to functional correctness.

## Portability

- Default async I/O path (`tokio`'s standard reactor) must work identically on Linux, macOS, and Windows. Do not introduce Linux-only dependencies into the core engine.
- `io_uring`-based optimization (`tokio-uring` or similar) is Linux-only and is a *future, optional* fast path relevant to the eventual agent/headless phase, not this one. Do not add it to the human-operator GUI codebase now — the concurrency profile doesn't need it here.
- No Linux-desktop assumptions (no GTK/Qt bindings, no specific display-server assumptions) — `winit` and `wgpu` already abstract this; keep it that way.

## Non-goals

- No adapters for LinkedIn, X, Indeed, or other bot-walled sites; no bot-evasion in the core engine.
- No pixel-parity chase with Chromium beyond what WPT compliance already gets you.
- No DRM/EME.
- No horizontal multi-node scaling concerns — this is single-binary, single-machine scope.
- The agent must not couple to a specific inference provider — Groq is one `InferenceBackend`, not a dependency of the agent logic. Do not commit the local-only development harness or any API keys.

**Research dispositions (2026-07, primary sources cited inline):**
- **B1 streaming parse→layout.** *Finding:* html5ever's `parse_document` returns a `Parser: TendrilSink` already exposing `process(chunk)`/`finish()` (push-incremental — docs.rs/html5ever); browsers additionally run a *secondary preload scanner* over raw bytes for `<img>/<link>/<script>` while the main parser blocks on sync scripts/CSS, and CSS is render-blocking so first paint is gated on `<head>` CSS (web.dev/preload-scanner; MDN speculative parsing). *Disposition:* **reuse** html5ever's incremental sink; **hand-roll** the first-paint checkpoint (small engine glue); **defer** the preload scanner until external-resource fetch (D4) exists. Prereq: a chunked-body stream in `engine/net` (today `BodyExt::collect` buffers the whole body). *Roadmap:* the §1 "Click-to-navigate latency" bullet's "stream socket bytes into the html5ever sink" resolves to: net yields body chunks → `parser.process()` per chunk → first-paint checkpoint after `<head>` CSS → incremental body layout. *Pass-2 resolution (2026-07) — the three open pieces closed:* **(1) Net chunked-body stream API:** consume the response `Incoming` body via **`http_body_util::BodyExt::frame().await`** per chunk (each `Frame` is a `Bytes` data chunk) — *not* `collect()` — and feed each chunk to `parser.process(chunk)`; `async-compression` (B3) wraps the chunk stream *before* the parser for `Content-Encoding` (docs.rs/http-body-util BodyExt). **(2) First-paint checkpoint:** DOM construction is incremental, but the initial render is **render-blocking on CSS** (inline `<head>` `<style>` + `<link>`) *by design, to prevent FOUC* — so the checkpoint = **once `<head>` and its render-blocking CSS are processed (CSSOM ready), do the first layout+paint of the DOM-so-far (at `<body>` start); thereafter re-layout/re-paint incrementally per streamed chunk via A2** (web.dev critical-path/FCP; MDN critical-rendering-path). **(3) Net redesign (folds in B2's connector):** replace the manual `conn::http1::handshake` with **`hyper_util::client::legacy::Client` + `hyper-rustls::HttpsConnector`** (`HttpsConnectorBuilder`, ALPN `h2,http/1.1`) — one stack that unifies **pooling/happy-eyeballs (B2) + h2 (B3) + a streamable `Incoming` body via `.frame()` (B1)**; the Client is cheap-to-clone with a reused pool (docs.rs/hyper-rustls, hyper-util `legacy::Client`). *Disposition:* **reuse the whole stack** (`hyper-util` Client + `hyper-rustls`), stream via `.frame()`, **hand-roll** the render-blocking-CSS-gated first-paint checkpoint; incremental relayout is A2; preload scanner stays deferred on D4. No WPT target (perf/architecture).
- **B2 connection reuse / preconnect / DNS-prefetch.** *Finding:* `hyper_util::client::legacy::Client` already provides a keep-alive connection **pool** (default 90s idle), and its `HttpConnector` does **Happy Eyeballs** (RFC 6555/8305, 300ms default), threaded DNS, and `SO_KEEPALIVE` (docs.rs/hyper-util). Chrome reaps idle *preconnected* sockets at ~10s and warns that over-preconnecting saturates the pool; preconnect leaks navigation intent (DNS+TCP+TLS) so it must be user-initiated and same-origin-biased, with `dns-prefetch` as the cheaper maybe-origin fallback (developer.chrome.com/Lighthouse, web.dev/resource-hints, MDN). *Disposition:* **reuse** hyper-util `legacy::Client` + `HttpConnector` for pooling/happy-eyeballs/DNS, wrapped with a rustls TLS connector (**reuse** `hyper-rustls`, small dep, or a tiny custom `tower::Service`) — replaces the current per-request `conn::http1::handshake`; this is also compatible with B1's streaming body (`Incoming`). **Hand-roll** the speculative preconnect trigger in the shell (hover/pointerdown → warm a pooled socket, ~10s idle cap, bounded concurrency), gated user-initiated; **defer** it until the pooled client + a shell hover signal exist. *Privacy constraint:* no unsolicited cross-origin preconnect without policy. *Roadmap:* resolve the §1 "connection reuse/pooling" and "speculative preconnect/DNS-prefetch" clauses to the above. *Pass-2 note (2026-07): the connector question is closed under **B1's pass-2 resolution** — use `hyper-rustls::HttpsConnector` (ALPN `h2,http/1.1`) with `hyper_util::client::legacy::Client`; one stack unifies pooling (B2) + h2 (B3) + streamable body (B1). Prefer `hyper-rustls` over a custom `tower::Service`.* No WPT target.
- **C1 realize hibernation.** *Finding:* Chromium splits it into two mechanisms that map onto our tiers — **freeze** (Energy Saver: after a tab is hidden+silent ~5min, stop JS/CPU and throttle DOM timers to ~1/min via "intensive wake-up throttling"; **keeps full RAM**; dispatches `freeze`/`resume`; audio/WebSocket/RTC tabs exempt) and **discard** (Memory Saver: unload the tab — resident drops from ~80–300MB active to ~5–10MB metadata/favicon, reloads on activation; this is the real RAM reclaim) (chromium.org tab-discarding, developer.chrome.com freezing/energy-saver). SpiderMonkey uses **compartments** for per-tab isolation (Firefox's own model) and **realms** for globals, loading self-hosted code once from a single compressed stream at init — there is **no V8-style deployable startup snapshot** (firefox-source-docs, spidermonkey.dev). *Disposition:* **hand-roll** the freeze/discard actions in the compositor (tiers exist, actions don't): background-CPU tier = *freeze* (throttle timers/JS task queue, keep DOM+layout, shrinking GC), hibernated tier = *discard* (drop fragment tree + rasterized tiles + the tab's realm, retain URL/DOM-source, relayout/re-fetch on wake — where RAM is actually reclaimed). **Reuse** SpiderMonkey compartment/realm-per-tab over the shared process-global runtime for low per-isolate baseline. *Reframe:* the §1 phrase "share SpiderMonkey startup snapshots across isolates" is inaccurate — replace with "compartment/realm-per-tab over one process-global runtime (SpiderMonkey has no V8 snapshot; self-hosted code loads once at init)." *Prereq:* per-tab baseline-RSS measurement (§8) to prove the freeze/discard deltas. No WPT target.
- **A1 Vello GPU paint tier.** *Finding:* the tier crates exist — full `vello` (GPU compute) and the "sparse strips" `vello_cpu` / `vello_hybrid` — but as of early 2026 sparse strips is **0.0.7, experimental, no API-stability guarantee** (any release likely breaking). `vello_cpu` is the most mature (near-beta, fastest Rust CPU renderer, now has Gaussian blur / drop-shadow / flood, non-layer clipping, gradients, opaque-image fills); `vello_hybrid` (GPU) and full `vello` remain alpha with known rough edges (complex-paint glyphs, high-winding paths, WebGL-on-Adreno, conflation). Even **Servo adopted Vello only for the bounded `<canvas>` surface, keeping WebRender for page compositing** (linebender.org TMIL blogs; github.com/linebender/vello releases; doc.servo.org/vello_cpu). *Disposition:* **defer** the focused-tab GPU tier (`vello`/`vello_hybrid`) on API instability + alpha rough edges — the existing `Painter` trait isolates the swap, so waiting is cheap and correctness-first. **Keep `tiny-skia`** as the CPU tier now; **track `vello_cpu`** as the eventual *richer* CPU tier (adds blur/shadow/clip we lack) and **reuse** it at its first stable (≥0.1) release; if prototyping the GPU tier sooner, pin an exact 0.0.x. *Roadmap:* §1 "Frame timing + rendering quality" resolves to — `Painter` seam ready → adopt `vello_cpu` at first stable release → GPU `vello_hybrid` after API stability + glyph fixes; `tiny-skia` correctness-first meanwhile. *Pass-2 freshness (2026-Q1):* Vello CPU is now "broadly usable / ready for production use cases" **feature-wise**, but its **API is still explicitly not stable** (not ≥0.1); Vello Hybrid is "roughly beta" with no API-stability guarantee; sparse strips still 0.0.7 (linebender.org tmil-25). Disposition **unchanged (defer)**, trigger **sharpened**: the CPU tier's *features* are ready, so **API stability is now the sole blocker** — adopt `vello_cpu` behind the `Painter` seam the moment its API stabilizes. No WPT target (rendering quality → visual reftests, not WPT layout/DOM).
- **A2 incremental / damage-driven relayout + repaint.** *Finding:* the established model, used across engines, is the **double dirty-bit** — a node dirty bit (`IS_DIRTY`) plus an ancestor **summary bit** (`HAS_DIRTY_DESCENDANTS`) so incremental layout skips any subtree whose summary bit is clear — combined with **RestyleDamage**: diff old vs new computed style into damage levels (Repaint vs Reflow vs box reconstruct), with forced reflow on viewport change (servo wiki Layout-Overview / Workweek-incremental-layout; developer.chrome.com BlinkNG; arXiv:2411.10659 "Spineless Traversal" as a newer refinement). **`taffy` supports incremental relayout**: a per-node cache + `mark_dirty()` invalidates a node and its ancestors so unchanged subtrees reuse cached results — it is *not* full-recompute-only; Servo is actively reusing taffy boxes incrementally (servo/servo PR #37751). *Disposition:* **reuse the algorithms, hand-roll the engine glue** (they're algorithms, not crates): add per-node dirty + summary bits and a Servo-style RestyleDamage diff over our `ComputedStyle`, feeding the compositor's existing `Damage` for repaint; **reuse** taffy's `mark_dirty`+cache for flex/grid subtrees, hand-roll the equivalent cache for our block/inline fragment tree. **Defer, blocked on** dynamic mutation existing (D3 DOM mutation / scroll-hover restyle / resize) — the current one-shot static render has nothing to invalidate; but carry the dirty/damage bits in the fragment tree **now** so the incremental path is a fill-in, not a rewrite. *Pass-2 resolution (2026-07) — the three data structures, beyond the double-dirty-bit:* **(1) Invalidation (skip re-matching):** build a **RuleFeatureSet** at stylesheet-parse time mapping each key (class / id / attribute) → **descendant vs sibling `InvalidationSet`s**; a runtime class/id/attr change looks up only those keys and marks matching descendants/siblings dirty, with a **whole-subtree fallback** for complex selectors (chromium blink style_engine / RuleFeatureSet). **(2) Style sharing (skip re-computing):** a **matched-properties cache** (hash the matched declarations → reuse a cached `ComputedStyle`) + **sibling style sharing** (reuse a neighbor's computed style when inputs are identical), honoring the documented **sharing-breakers** (id, inline style, container units, registered event handlers) (webkit matched-declarations cache). **(3) Layout reuse (skip re-layout/rebuild):** a **`LayoutDamage` taxonomy** — box-tree *structural* rebuild → box rebuild *reusing structure* → *fragment-layout-only* ("empty" damage) → repaint-only — with **cached fragments on `LayoutBox`** invalidated precisely in the damage traversal; Servo's active 2025 incremental-layout PRs are the live reference (servo/servo #37751, #37957 "skip box tree construction", #38084 inline, #37866/#37868 atomics/floats). *Disposition:* **reuse all three algorithms, hand-roll the engine glue** over our arena DOM + from-scratch block/inline fragment tree (RuleFeatureSet in `manuk-css`, matched-properties/sharing cache in the cascade, `LayoutDamage` enum + cached fragments on `LayoutBox` → the compositor `Damage`). Still **deferred, blocked on D3 dynamic mutation**, but these are the structures to seed now. *Roadmap:* §1 "Incrementality" resolves to the double-dirty-bit + RestyleDamage → `Damage` design above, plus the RuleFeatureSet / style-sharing / LayoutDamage structures here. No new WPT target (correctness reuses the same layout reftests, run incrementally vs full).
- **A3 shaped-run + decoded-image caching.** *Finding:* Firefox/Blink both cache shaped runs; the standard granularity is **word-level** (split on word boundaries) for hit rate, with the known caveat that per-word shaping breaks OpenType **contextual** features needing cross-word context — so word-level needs a run-level fallback for such scripts/features (bugzilla 761442; harfbuzz shape-plans keyed on features). Cache key components: **font identity + size + run text + script/direction/lang + features**. Decoded-image caches use **LRU + a byte budget over discardable memory** (freeable under memory pressure; Chromium `cc/tiles/image_decode_cache.h` + Skia discardable, timestamp-LRU eviction). *Disposition:* **hand-roll the keys + per-tab accounting, reuse `lru` for eviction** (LRU is a tiny solved crate; keys/accounting are engine-specific). (1) **Shaped-run cache in `manuk-text`** extending the existing `FontKey` font cache — key `(FontKey, quantized size, text, script/dir/features)`, word-level with run-level fallback (safe now: our shaping is Latin/no-features), `lru` + byte budget — **buildable now**, a low-risk win especially once A2 re-shapes on relayout. (2) **Decoded-image cache** keyed `(resource-id/URL, decode size, format)` → RGBA, LRU + byte budget — **defer, blocked on** image loading/decoding (D4). *Cross-ref C1/§1 per-tab memory:* account both caches **per tab** and wire to the compositor budget so a discarded/hibernated tab drops them. (HTTP cache is separate — net/loading.) *Roadmap:* §1 "Caching" resolves to shaped-run-now + decoded-image-on-D4, both per-tab-accounted and evicted on C1 discard. No WPT target.

- **E2 local encrypted password store + autofill — PASS-2 RESOLVED (security-critical; audited-crate orchestration only, zero hand-rolled crypto).** *Finding:* Chromium's model = a random key in the **OS secret store** encrypts the DB at rest — Windows **DPAPI**, macOS **Keychain** (16-byte key → PBKDF2), Linux **Secret Service/libsecret** (with a weak `v10` hardcoded-password *fallback* when no keyring). Autofill matches by **origin (scheme+domain required)**; same-site sharing (`www`↔`m.example.com`) is scoped by the **Public Suffix List (eTLD+1)**; cross-origin-iframe credential leakage is a real past vuln class; Apple maintains shared `password-manager-resources` (domain-equivalence quirks) (chromium.org os_crypt / os-x-keychain; developer.chrome.com identity; github.com/apple/password-manager-resources). *Disposition:* **reuse audited crates ONLY, never hand-rolled crypto** — `keyring` (wraps DPAPI/Keychain/Secret Service), a `RustCrypto`/`ring` AEAD (AES-256-GCM or ChaCha20-Poly1305) for at-rest with the OS-store key, `psl`/`publicsuffix` for eTLD+1 scoping, reuse Apple's resource list for quirks; hand-roll only orchestration + origin-matching policy. **Flagged pass-2** for the *policy* architecture (NOT crypto primitives): the Linux-no-keyring fallback stance (Chromium's weak default is a known problem — refuse? require an app primary password?), the exact origin-match rules (cross-origin iframe, HTTP↔HTTPS downgrade, PSL scoping — a wrong rule leaks credentials), and whether to require a primary password. *Pass-2 resolution (2026-07):* **Key management** — the primary encryption key lives in the **OS secret store via `keyring`** (v4; DBus Secret Service / macOS Keychain / Windows). When it is **unavailable** (headless / no Secret Service — Chromium's weak spot), fall back to a **primary-password-derived key via `argon2` (Argon2id, PHC winner)** — **never** Chromium's hardcoded-password fallback; the DB is sealed with a **RustCrypto AEAD (`chacha20poly1305` or `aes-gcm` — both NCC Group-audited, no significant findings)**; the KDBX4/KeePass format (Argon2id + AEAD) is a candidate audited on-disk format (github.com/open-source-cooperative/keyring-rs; RustCrypto/AEADs NCC audit; P-H-C/argon2). **Origin matching** — Chromium keys credentials by **`signon_realm` = scheme+host+port** (exact match = default fill); **PSL/eTLD+1 matching applies only to HTML forms** and is surfaced as a *related-domain* suggestion (not silent), via the `psl` crate; **no scheme-downgrade** (an HTTPS-saved credential is never auto-filled into HTTP); **cross-origin iframes** require the field's own origin to match (the documented leakage class — no silent cross-origin fill); affiliation (Digital Asset Links / Related Website Sets) is opt-in + verified via a well-known file → **defer** (chromium password_manager signon_realm/PSLMatchingHelper; developer.chrome.com seamless-credential-sharing). *Disposition:* **reuse audited crates only** — `keyring` + RustCrypto `chacha20poly1305`/`aes-gcm` (audited) + `argon2` + `psl`; **hand-roll only orchestration + the origin-match policy**; the constraint (no hand-rolled crypto primitives) is satisfied. *Roadmap:* §9 + cross-ref §7 Security; no WPT target.
- **E3 translate-page.** *Finding:* Firefox Translations = fully **local** on-device NMT (Bergamot/Marian → WASM, CPU-optimized), models **~20 MiB per language pair downloaded on demand** (not bundled → base binary stays lean) + inference runtime; entirely private (mozilla blog/hacks Bergamot; firefox-source-docs translations; mozilla/firefox-translations-models). In-page translation **block-segments** the DOM, flattens inline runs (`<a>/<b>/<i>`) with boundary markers, translates, and reinjects preserving structure. *Disposition:* **confident — reuse, two lean tiers, hand-roll the DOM glue.** (1) **BYO-endpoint (lead):** reuse **E6's `InferenceBackend`** to translate — zero model runtime, user's configured endpoint (privacy = user's choice). (2) **Local private tier:** reuse Bergamot/Marian models on-demand via FFI to `bergamot-translator` or ONNX (`ort`) — the local-runtime pick is a deferred sub-decision behind E6. DOM extraction/reinjection reuses our block/inline distinction (layout already separates them). *Roadmap:* §9 item, reuses E6; no WPT target.
- **E4 developer tools / console — PASS-2 RESOLVED (strategic: implement a WebDriver BiDi remote end, not CDP).** *Finding:* CDP is Chromium-only and **not a standard**; **WebDriver BiDi** is the **W3C** cross-browser automation standard (bidirectional/evented like CDP but interoperable) — Firefox ships it production-ready, Puppeteer & Playwright support it, and it is **WPT-tested** (`webdriver/tests/bidi/`, wdspec) with modules session/browser/browsingContext/script/network/log/input/storage (developer.chrome.com/blog webdriver-bidi; w3.org/TR/webdriver-bidi; web-platform-tests/wpt). *Disposition lean (validate in pass-2):* implement a **WebDriver BiDi subset**, **not CDP** — this makes the *agent* drivable by Puppeteer/Playwright (the two-birds) **and** is standards-aligned + WPT-verifiable (fits WPT ground-truth, avoids CDP's Chromium-quirk surface, itself a non-goal); a bespoke inspector only for the human DevTools *UI*. **Flagged pass-2:** the minimal viable module subset (session + browsingContext + script + network + log?), the implementation effort, and how BiDi's session/navigable model maps onto the existing `AgentBrowser`/`run_task` + `engine/net`. *Pass-2 resolution (2026-07):* BiDi is **JSON-RPC over WebSocket** — a *remote end* (server) = a WebSocket endpoint (**reuse `tokio-tungstenite`, already in our stack**) + a JSON command dispatcher matched against the spec's CDDL, with `session.subscribe` for events; `chromium-bidi` and Firefox's `remote/` are references (w3.org/TR/webdriver-bidi; github.com/w3c/webdriver-bidi; GoogleChromeLabs/chromium-bidi). The commands mostly **wrap operations we already have**: `browsingContext.navigate`→`AgentBrowser.navigate`, `browsingContext.captureScreenshot`→`screenshot_png`, `getTree/create/close`→tab model, `network.*` events→`engine/net`, `log.entryAdded`→console. **Minimal buildable-NOW subset (pre-D3):** `session` + `browsingContext` (navigate/captureScreenshot/getTree/create/close + load events) + `network` events + `log` — this alone makes the agent **Puppeteer 23+-drivable today** (the two-birds), reusing `AgentBrowser` + `engine/net`. **Deferred sub-commands:** `script.evaluate`/`callFunction` (blocked on D3 JS/DOM bindings), `input.performActions` (blocked on hit-testing + element refs — ties to G-c a11y tree). Moderate effort (WebSocket + JSON dispatcher + CDDL-shaped types); Puppeteer is more BiDi-ready than Playwright (still partial, microsoft/playwright#32577). *Disposition:* **build the BiDi remote-end subset above** on `tokio-tungstenite`. *New WPT target:* `webdriver/tests/bidi/`. *Roadmap:* §9 (+ §4 agentic two-birds).
- **E5 extensions.** *Finding:* the full WebExtensions surface is enormous (hundreds of APIs, content scripts, background workers, permissions) — a scope trap and a major attack surface. But the #1 high-value case (ad/tracker/content **blocking**) has a **declarative, code-free** path: Manifest V3's `declarativeNetRequest` replaced runtime `webRequest` interception with declarative URL-filter rules (safer, privacy-preserving, no host permissions), and Brave's **`adblock`** (adblock-rust) is a mature Rust crate — EasyList/EasyPrivacy/uBO-syntax network blocking + cosmetic filtering, tokenized matching (69× speedup), **designed to run inside a native browser core** (developer.chrome.com declarativeNetRequest; github.com/brave/adblock-rust; brave.com blog). *Disposition:* **confident, skepticism upheld — extensions-as-a-runtime is OUT OF SCOPE** (scope trap + attack surface; content scripts = arbitrary JS injection, excluded); password managers are native (E2), not extensions. The valuable subset is delivered **natively without any extension runtime**: **reuse `adblock`** in `engine/net` (+ cosmetic hooks into DOM/paint) consuming standard filter lists — this is a §2/§7 content-blocking feature, not "extensions." *Roadmap:* §9 rejects an extension runtime; add native content-blocking (`adblock`) to §2 loading / §7 privacy; no WPT target.
- **E6 bring-your-own-key in-browser AI — PASS-2 RESOLVED (security model: layered CaMeL structural separation + OWASP Action-Guard).** *Finding:* the panel is architecturally trivial — **reuse the `InferenceBackend` abstraction**; the headful AI panel and the headless agent are *the same core* with different front-ends (the project thesis). The crux is security: **indirect prompt injection** is a systemic, unsolved challenge for AI browsers — page content (incl. hidden white-on-white text, HTML comments, and **screenshot-borne "unseeable" injections**) gets fed to the LLM without separating user instructions from untrusted content, causing cross-site actions (fetch OTPs, banking) (brave.com agentic-browser series; simonwillison.net; arXiv:2511.19477). Anthropic's Claude-in-Chrome documents the mitigation stack: **per-site permissions** (first line of defense), **explicit confirmation for sensitive actions** (publish/purchase/share — retained even in autonomous mode), **system-prompt hardening**, and **classifier detection** of injections in untrusted content — reducing a challenge-set attack rate 35.7%→**0%** (anthropic.com/news/claude-for-chrome, /research/prompt-injection-defenses). *Disposition (pass-1):* **reuse `InferenceBackend` (confident); the security/consent model was the open gap → resolved in pass-2 below.** **Critically, the screenshot-injection vector applies to our EXISTING agent path** (which already sends screenshots) — so this is also a §4/§7 retrofit, not only a new panel. **Flagged pass-2:** per-site permission model, sensitive/cross-origin action taxonomy + confirmation UX, structural instruction-vs-page-content channel separation in `run_task`, and whether to add an injection classifier pass — for **both** the panel and the agent. *Pass-2 resolution (2026-07) — a layered architecture, all reuse of established patterns (no novel crypto/ML):* **(1) Structural (strongest) — dual-LLM / CaMeL:** the privileged planner sees only the **trusted user task** and emits a plan; **untrusted page content (incl. screenshot-derived text) is data with provenance**, processed by a quarantined path with **no tool access**; a taint/capability check gates every tool call — "structurally unable to influence flow, like parameterized queries for SQLi" (arXiv:2503.18813 CaMeL; simonwillison.net/2025/Apr/11/camel). **(2) Per-site permissions** (Anthropic's first line of defense). **(3) OWASP Action-Guard human-in-the-loop:** classify actions by **irreversibility heuristics** (financial / administrative / externally-visible / data-deletion / privilege-escalation / cross-origin credential use) and require explicit confirmation, **separating decision from execution** (genai.owasp.org LLM06/LLM08; AI Agent Security Cheat Sheet). **(4) Injection classifier** over untrusted content (Anthropic 35.7%→0%). **Concrete Manuk change:** retrofit `run_task` so the `Observation` (page text + screenshot) is untrusted, provenance-tagged data **kept out of the instruction/planner channel** (we currently merge it — the Comet flaw), and gate cross-origin navigate/click/form-submit behind capability + confirmation. **This hardens BOTH the future panel AND the existing agent path** (which already sends screenshots — the unseeable-injection vector). *Disposition:* **reuse `InferenceBackend` (arch) + the layered CaMeL/permissions/Action-Guard/classifier defense (security)**; the classifier runs via the existing backend. *Roadmap:* §9 (+ §4 agent, §7 security). *No WPT target.*
- **E7 profiles / containers / VPN.** *Finding:* Firefox distinguishes **profiles** (complete separation of *all* data — bookmarks/passwords/settings/extensions/history/cookies) from **containers** (Multi-Account: separate only cookies/logins/site-data *within* a profile — a contextual-identity cookie-jar partition), with **Total Cookie Protection** = a per-website cookie jar (partitioned storage) (support.mozilla.org containers/profiles/TCP). SOCKS5/HTTP proxy is application-level, unencrypted routing; `tokio-socks` is the mature async Rust client; `wireproxy` even exposes WireGuard *as* a SOCKS5 proxy — so "VPN" can reduce to "point at a proxy" (github.com/sticnarf/tokio-socks; github.com/windtf/wireproxy). *Disposition:* **confident, lean framing.** Profiles/containers = a **storage-partition** design (profile ⊃ container jar ⊃ per-site jar/TCP) — **hand-roll the partition key** over the storage layer, but **defer, blocked on a storage layer** (we have none: no cookies/cache/history/localStorage yet). **VPN = reuse SOCKS5/HTTP proxy** (`tokio-socks`) so the browser routes through a user-provided proxy (WireGuard via `wireproxy` externally); **bundling a WireGuard/OpenVPN client is out of scope** (scope-heavy, no strong reason) — the RESEARCH-recommended lean framing. *Roadmap:* §9 (profiles deferred on storage; SOCKS proxy in §2 net); no WPT target.

### 10. Blue-ocean capabilities — Axis G

**Research dispositions — Axis G pass-1 baseline (2026-07, 2 searches/item). Assessed: real utility vs demo, engine surface needed, uniquely enabled by our shared-core architecture.**
- **G-a unified human+agent session.** *Finding:* NO LONGER novel — human↔agent live-session handoff is a crowded 2025–26 space: Tandem Browser ("AI in the browser itself, same tabs/cookies/session, explicit handoffs, tab locks"), Gemini-in-Chrome auto-browse, chrome-use, browser-act (cross-device handoff) all ship take-over-anytime in the user's live session (github.com/hydro13/tandem-browser; blog.google gemini-chrome). *Disposition:* **confident — real utility, architecturally natural on our shared `engine/page` core (reuse engine/page + the agent core), but NOT blue-ocean** (market caught up). Pursue as a low-marginal-cost feature of the shared-core design, not a moonshot; the *architectural* cleanliness (both front-ends on one core, deterministic handoff) is the only differentiator vs CDP/Electron bolt-ons. *Roadmap:* §10 natural feature; no WPT target.
- **G-b local-first semantic history — PASS-2 RESOLVED (stack is off-the-shelf; the storage tension is solved by LEANN-style recompute).** *Finding:* on-device semantic search is active research (LEANN low-storage index; local embedding models; ~7× embedding storage blowup is the key constraint) but a **browser-integrated private semantic index of *actually-read* page content** — uniquely enabled by our engine already parsing every page to clean text + the local BYO-AI (E6) — is **not a mainstream feature** (browser history is keyword-only) (arXiv:2506.08276 LEANN; github vector-search). Volatility scoring is grounded: per-URL adaptive re-embed interval by observed change rate (code-docs ~0.01%/day → weeks; news ~80%/hour → minutes) + content-addressing for dedup/change-detection. *Disposition:* **warrants deeper research** — genuinely differentiated and rides on surface we build anyway (clean text + local AI). **Flagged pass-2:** the vector-store choice (LEANN-style / `usearch` / sqlite-vec), the local embedding model, the storage budget vs §3 lightweightness (7× blowup), the volatility-scoring policy, and index privacy/encryption. *Pass-2 resolution (2026-07) — the whole stack is off-the-shelf, and the storage tension is solved:* **Embeddings** — reuse **`fastembed`** (Rust, ONNX, no-Tokio) with a **small quantized** model (BGE-small-Q / MiniLM), downloaded once then offline/on-device/private (github.com/anush008/fastembed-rs). **Vector index** — reuse an embedded store: **`sqlite-vec`** (usearch-backed HNSW, single-file, no server — the lean default) or LanceDB (github.com/asg017/sqlite-vec; lancedb). **The §3 storage tension is resolved by `LEANN`** (MLsys2026, arXiv:2506.08276): store only a **pruned proximity graph** and **recompute embeddings at query time** with the same encoder → **~97% storage savings / >50×**, <5% overhead vs raw, 90% top-3 recall, <2s search — and since we *already* have the encoder (`fastembed`) and re-parse every page, recomputation is natural. **Volatility** — per-URL adaptive re-embed interval by observed change rate + content-addressing for dedup/change-detection (pass-1). **Privacy** — the index is 100% local; **encrypt at rest reusing the E2 AEAD stack** (`chacha20poly1305` + OS-store/primary-password key). *Disposition:* **reuse — off-the-shelf mature crates + one research technique (LEANN) that resolves the lightweightness conflict.** Uniquely enabled by our parsed-clean-text + local BYO-AI (E6) + local-first ethos; the "stronger than expected" read holds and is now de-risked. *Roadmap:* §10 (+ ties E6, §3, and the E2 AEAD). No WPT target.
- **G-c agent-assisted accessibility.** *Finding:* the "one investment, two payoffs" thesis validates strongly. **`accesskit`** is a mature Rust cross-platform a11y bridge (Windows UIA / macOS NSAccessibility / Linux AT-SPI; tree schema modeled on Chromium's a11y abstraction; used by egui). Independently, web-agent research (WebArena et al.) uses the **accessibility tree** as the *preferred* observation channel — "a subset of the DOM with role/text/properties, structured yet more compact than DOM," read via semantic roles/labels "rather than raw pixels or full DOM" (github.com/AccessKit/accesskit; arXiv:2307.13854 WebArena; arXiv:2508.04412). *Disposition:* **confident — high-leverage, build ONE a11y/semantic tree** (DOM + ARIA/implicit roles + layout geometry): **reuse `accesskit`** for the screen-reader platform bridge; **hand-roll** the DOM→tree mapping (WAI-ARIA + HTML-AAM). Two payoffs: (1) human screen-reader accessibility; (2) the agent's **structured observation channel** (§4/D3 — better than raw text+screenshot, and reduces reliance on the injection-prone screenshot channel, cross-ref E6). This is engine surface D3 needs anyway → sequence together. *Roadmap:* §10 (+ D3, §4, E6). *New WPT target (partial):* `wai-aria/` for role computation.
- **G-d deterministic replay / provenance.** *Finding:* the established technique for reproducible agents is an **append-only event log recording model + tool responses** (models aren't reproducible, so you record their outputs), with strict-mode event-by-event divergence checking as a *proof* of reproducibility, and "decision-complete provenance" enabling re-evaluation of historical runs under new policy without re-execution (arXiv:2605.21997 "The Log is the Agent"; arXiv:2505.17716 record&replay). Crucially, **GPU rendering and layout are documented sources of replay-nondeterminism** (WebKit "Web Replay"; GPU kernel autotuning) — so GPU/cloud-VM agents diverge on replay. *Disposition:* **confident — real utility (auditable/reproducible agent runs for compliance/research/debug/regression), uniquely well-positioned.** Our agent is **display-free with a deterministic CPU raster** (`tiny-skia`) → stable screenshot hashes make exact replay achievable where GPU agents can't; the clean `InferenceBackend` seam records model I/O. **Reuse** the existing `AgentOutcome` transcript → extend to a full event log; low marginal cost. *Roadmap:* §10 (+ §4 agent, §5 correctness — replay doubles as a regression harness). No WPT target.
- **G-e instant per-tab resource honesty.** *Finding:* real pain (the founding "why is my fan spinning") but **not novel** — Chrome/Firefox already ship per-tab/per-process task managers + `about:memory`. The genuine wrinkle is **ours specifically**: our **isolate-per-tab-in-shared-process** model (C1) means the OS won't hand us per-tab CPU/RAM for free (unlike Chrome's process-per-tab). SpiderMonkey **does** support per-compartment/zone memory attribution (Firefox's `about:memory` reports per-JS-compartment; every GC cell belongs to ≤1 compartment) — but with a caveat that ~30–45% lands in "heap-unclassified" (imperfect attribution) (firefox-source-docs js/gc; blog.mozilla.org nnethercote). *Disposition:* **confident — do it, reuse SpiderMonkey per-compartment memory reporters** for per-tab JS heap + **hand-roll** compositor per-tab layout/paint/tile accounting + a shell task-manager UI. It rides on the same instrumentation **C1 + §8** need; the shared-process **self-attribution** (esp. per-tab *CPU*, which needs self-timing with no per-process split, and the heap-unclassified gap) is the wrinkle to solve there, not a blocker. *Roadmap:* §10 (+ C1, §8, §1). No WPT target.
